# Governance & Access Control

ToolJet's access control model operates at three levels: who can use an app, what they can see inside it, and what they can do within the workspace. This section covers the full permission model — from built-in roles to component-level rules — and how it integrates with your IdP.

## Built-in Roles

Every workspace user is assigned one of three built-in roles:

| Role | What they can do |
|---|---|
| **Admin** | Manage users, groups, data sources, workspace settings, and all apps. Full access to everything in the workspace. |
| **Developer** | Create, edit, and publish apps. Connect data sources. Cannot manage users or workspace-level settings. |
| **Viewer** | Use published apps. Cannot create, edit, or access the builder. |

These roles define the ceiling of what a user can do. App-level and group-level permissions can restrict further but cannot grant more than the role allows.

## App-Level Access Control

Individual apps have their own access settings independent of workspace role:

- **Private**: visible only to users explicitly granted access
- **Workspace**: accessible to all workspace members with Viewer role or above
- **Public**: accessible without authentication (for external-facing tools)

Per-app access can be scoped to specific groups or individual users. An app can be visible to the "Support" group but hidden from "Finance" even though both are in the same workspace.

## Component-Level Access Rules

Within an app, individual components can be conditionally shown, hidden, or disabled based on the current user's properties.

Any component's `Visibility` and `Disabled` properties accept `{{ }}` expressions. Common patterns:

- `{{ currentUser.groups.includes('admin') }}` — show an action button only to admins
- `{{ !currentUser.groups.includes('editor') }}` — disable a form field for read-only groups
- `{{ currentUser.email === row.owner }}` — show controls only to the record owner

This means a single published app can serve multiple roles with different UI surfaces — no duplicate apps required.

## Page-Level Access Control

Multi-page apps can restrict individual pages to specific groups. Users without access to a page are blocked at the navigation level — the page is not rendered, not just hidden in the menu.

Combined with component-level rules, this allows a single app to function as a unified tool for admins, editors, and viewers with fully differentiated experiences.

## Admin API

Workspace user and group management is available via REST API. Use it to:

- Programmatically provision users from an external HR system
- Sync group membership from a source of truth outside your IdP
- Automate onboarding workflows that create ToolJet users as part of a broader account setup flow
- Audit user and group state from external monitoring tools
