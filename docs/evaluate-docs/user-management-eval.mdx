# User & Workspace Management

This section covers how users are authenticated, provisioned, and organized in ToolJet. For enterprise deployments, the key questions are: Can ToolJet integrate with your existing IdP? Can user lifecycle be automated? How do workspaces map to your team structure?

## Enterprise SSO in Practice

When SSO is configured, users authenticate through your IdP and are issued a ToolJet session. No ToolJet credentials are created or stored.

Key behaviors:
- **Just-in-time provisioning**: users who authenticate via SSO for the first time are automatically created in ToolJet with the default group assignment you configure
- **Group sync**: IdP group membership maps to ToolJet groups at each login — see [Governance & Access Control](/evaluate/governance) for how group-based permissions work
- **Single sign-out**: ToolJet supports IdP-initiated logout, so users who are deactivated in the IdP lose ToolJet access at their next authentication attempt
- **Multiple IdPs**: a single ToolJet workspace can support multiple SSO configurations simultaneously (e.g., one for internal employees, one for contractors)

## SCIM Provisioning

SCIM 2.0 support allows your IdP to manage the full user lifecycle in ToolJet automatically.

| Event in IdP | Result in ToolJet |
|---|---|
| User created and assigned to ToolJet app | User account created, assigned to mapped groups |
| User group membership changed | ToolJet group assignment updated at next sync |
| User deactivated or removed from ToolJet app | ToolJet account suspended immediately |
| User re-enabled | ToolJet account re-activated with prior group assignments |

Supported IdPs for SCIM: Okta, Azure Active Directory, and any IdP that implements SCIM 2.0.

## Workspace Structure

A ToolJet workspace is the organizational unit: it contains users, apps, data sources, groups, and settings. Think of it as a tenant boundary.

**When one workspace is enough:** Most organizations run a single workspace for their entire internal tooling platform. Users are segmented by group, and apps are scoped to the groups that need them.

**When multiple workspaces make sense:**
- **Business unit isolation**: teams with no overlap in apps or users that need separate admin boundaries
- **External partner workspaces**: a workspace for contractors or external teams with controlled access to a limited app set
- **Environment separation**: a dedicated workspace per environment (dev/staging/prod) for strict separation — though this is typically handled by multi-environment support within a single workspace for most teams

Multiple workspaces run on a single ToolJet instance. Users can belong to multiple workspaces and switch between them. Data sources, apps, and groups are not shared across workspaces.
