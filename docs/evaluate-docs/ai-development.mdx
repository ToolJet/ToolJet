# AI Development

ToolJet has two distinct AI layers: one that helps developers build apps faster using natural language, and one that connects external AI coding agents to a live ToolJet instance via a standard protocol. This section covers what each layer can do, where it adds real leverage, and what still requires manual work.

## AI-Powered App Generation

ToolJet's built-in AI can scaffold a working application from a natural language prompt. Describe what you want to build — an approval queue, a data table with filters, a customer lookup form — and the AI generates an initial layout with components placed and configured.

**What it produces:** A functional app skeleton with component properties configured, event handlers set up, and visual structure reflecting your description. You connect real data sources, adjust queries to match your schema, and refine the UI for your use case.

**What it doesn't do:** The AI doesn't access your database schema or infer column names. It doesn't write complex business logic or generate production-ready queries. It's a head-start, not a finished build.

## The MCP Bridge

ToolJet exposes a Model Context Protocol (MCP) server that allows external AI coding agents to interact with a live ToolJet instance programmatically.

Agents like Cursor, GitHub Copilot, Claude Code, Cline, and Windsurf can — within a conversation — create components, configure queries, modify layouts, and update app settings by calling ToolJet's MCP tools directly.

## When to Use AI Generation vs. Manual Building

| Scenario | Recommended approach |
|---|---|
| Prototyping a new tool quickly | AI generation for initial scaffold, then manual refinement |
| Complex app with a specific data model | Manual building — the AI won't know your schema |
| Repeating patterns across multiple apps | Manual + Modules for reusable parts |
| Team already in an AI coding environment | MCP bridge for programmatic changes |
| Business users making UI changes | Manual — AI generation is a developer tool |
