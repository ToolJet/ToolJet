# Deployment & Infrastructure

ToolJet runs wherever your infrastructure is. This section covers the supported deployment targets, what a self-hosted production architecture looks like, the external dependencies, and what ongoing operations entail. If you have data residency requirements, network isolation constraints, or an existing container platform, this is where to evaluate whether ToolJet fits.

## Resource Requirements

These are baseline figures for a production HA deployment. Actual requirements depend on user count, query volume, and workflow frequency.

| Component | Minimum | Recommended (production) |
|---|---|---|
| App Server | 2 vCPU, 2 GB RAM | 4 vCPU, 4 GB RAM × 2 replicas |
| Worker | 1 vCPU, 1 GB RAM | 2 vCPU, 2 GB RAM × 2 replicas |
| PostgreSQL | 2 vCPU, 4 GB RAM | Managed service (RDS db.t3.medium or equivalent) |
| Redis | 1 vCPU, 1 GB RAM | Managed service (cache.t3.micro or equivalent) |

## Upgrades

ToolJet releases follow semantic versioning. The upgrade process is:

1. Pull the new Docker image version
2. Run database migrations (included in the image, auto-runs on startup)
3. Rolling restart of app server and worker replicas

Migrations are backwards-compatible within minor versions. Major version upgrades include a migration guide. The Kubernetes Helm chart supports rolling upgrades with zero downtime for the app server.

## Secrets Management

ToolJet reads all configuration from environment variables. Integration with secret managers (AWS Secrets Manager, HashiCorp Vault, Kubernetes Secrets, Azure Key Vault) is handled at the deployment level — inject secrets as env vars at container startup. ToolJet does not require direct access to a secrets manager.
