# App Builder & Components

ToolJet's app builder is a visual development environment for building data-connected internal tools. This section covers what the builder can produce, how the component library is structured, and where the visual model hits its limits.

## Data Binding

Components connect to query results through dynamic expressions. Any component property that accepts a value can be bound to a query output, component state, or page variable using `{{ }}` syntax.

- A table's data prop accepts `{{ queries.getUsers.data }}`
- A button's disabled state accepts `{{ currentUser.groups.includes('viewer') }}`
- An input's default value accepts `{{ components.dropdown1.value }}`

This means components re-render reactively as query results and user interactions change app state — without writing render logic.

**Events** drive interactivity: button click, form submission, page load, query success, and query failure all emit events that can trigger queries, navigate pages, open modals, show notifications, or set variables. Multiple actions can be chained on a single event.

## Multi-page Apps

Applications can have any number of pages. Each page gets a URL slug, an optional icon, and configurable access control.

- **Navigation menu** is configurable as top or side, with Text, Icon, or Text + Icon display
- **Page access** can be open to all users, restricted to specific users, or gated by user group
- **Nav items** can point to app pages, external URLs, or other ToolJet apps in the same workspace
- **Hidden pages** can still be reached via the Switch Page event or direct URL

## Custom Code

When visual configuration isn't enough, JavaScript and Python queries run directly within the app context.

Custom code queries have full access to component values, other query outputs, and page variables. Common uses:

- Transform or reshape query results before binding to a component
- Compute derived values from multiple inputs
- Conditionally trigger other queries based on logic
- Directly set or reset component values

No separate backend required — the code runs in the app context with access to the full app state.

## Modules

Modules are reusable UI bundles: components, queries, and logic packaged together and published to your workspace. They accept typed inputs and return outputs, so the same module behaves differently depending on where it's used.

Updates to a module propagate automatically to every app that uses it. Teams use modules for shared patterns like customer profile panels, approval forms, filter bars, and navigation components.
